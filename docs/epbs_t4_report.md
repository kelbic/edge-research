## 2. Свод операционных правил для monad-liquidator

Маркер: mainnet-USDe трек (газ-модель, выбор swap-пути) пересчитать **до
активации** Glamsterdam; Base не трогать до соответствующего OP-stack-форка.

1. **(i) Роутинг**: толстые агрегаторные calldata-пути на L1 исключить из
   выбора маршрута (20KiB → +150%, floor биндится); компактный роутинг и
   5KiB-класс остаются ~0% в SFI.
2. **(ii) Пре-сид**: заранее засеять балансы/слоты эксекьютора по целевым
   токенам — первый приём токена стоит +77,920 gas/слот в SFI (+88,820 в CFI).
   Операционная дисциплина, не капитал.
3. **(iii) Дроп access-lists**: листинг ключа (3,948) дороже холодного доступа
   (2,100) на 1,848; полный чистый штраф с учётом остаточного warm-доступа
   **+1,948/ключ, +1,180/адрес** (в CFI: +2,148/ключ). Для типового AL
   (2 адреса + 4 ключа) хранение AL = +10,152 gas (+3.1% к SFI-стоимости
   ликвидации). До форка AL по-прежнему экономит ~100/ключ — переключение
   строго по активации.
4. **(iv) Headroom**: gas limit tx обязан покрывать `intrinsic + 64·байт
   calldata (+ data-цена AL)` даже когда ожидаемый gasUsed ниже — иначе tx
   невалидна. Для 20KiB минимальный лимит 1,331,720 (сегодня хватало ~533k);
   проверку добавить в подачу ДО активации.

## 4. Расхождения плана с EIP-текстами (зафиксировано явно)

1. **«+1,848 к холодному» (план §1, AL-правило)**: по параметрам EIP-7981
   это стоимость *листинга* ключа против холодного доступа (3,948−2,100=1,848),
   без остаточного warm-доступа, который всё равно оплачивается. Полный чистый
   штраф = **+1,948/ключ** (1,900+2,048+100−2,100). Числа согласованы, разница
   методологическая; `al_rule_7981()` считает обе величины. Вывод «AL мёртв»
   не меняется.
2. **«интринсик 21k вниз» (EIP-2780)**: в интернет-версии это не единый
   срез, а декомпозиция: вызов контракта без value = 15,000; ETH-перевод
   существующему EOA остаётся 21,000; создание аккаунта переезжает в
   state-gas 183,600. В модели взято 15,000 для наших паттернов
   (вызов контракта).
3. Остальные несущие числа плана (64 gas/байт; 97.9k/183.6k; ~4.9×; +77.9k;
   +150% на 20KiB) совпали с первоисточниками точно.
4. Замечание вне модели: план §0 упоминает «devnet-7 конфиг смержен 10.07»,
   но в `network-configs/` master-ветки glamsterdam-devnets на 2026-07-12
   виден только devnet-6 — датчик S7 поймает появление каталога devnet-7.

## 5. Что НЕ удалось верифицировать по первоисточникам ([S]-допущения модели)

- **Доля нулевых байт calldata = 0.5** — допущение Gate 0 (ABI-паддинг);
  якорит today-стоимость 20KiB-паттерна = 533k из pre-registration. При
  100% ненулевых байт ΔSFI(20KiB) была бы ≈ +58%, а не +150% — чувствительность
  дельты к составу calldata существенна, реплей на тестнете обязан мерить
  фактический состав.
- **Стейт-профили exec** (ликвидация: 8 cold-аккаунтов, 25 cold-слотов,
  10 storage-обновлений; своп: 4/12/6) — синтетические оценки, влияют только
  на CFI-ветвь (8038); SFI-дельты от них не зависят (floor/слоты/AL).
- **exec_gas 300k** для свопов — предположение плана (§1: «300–400k»); floor
  5KiB (327,680) начал бы биндиться при exec ниже ~276k (327,680 − calldata
  51,200) — запас у допущения небольшой.
- Параметры 8038 помечены в самом EIP как «not yet final» (Draft) — вся
  CFI-ветвь условна по определению; числа заморожены в PARAMS-конфиге и
  правятся одной строкой.
