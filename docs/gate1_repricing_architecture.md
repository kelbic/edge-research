# Gate 1 — замер латентности репрайсинга Nexus Mutual (архитектура)

## Вопрос, на который отвечаем

Тезис edge (STATE.md, Направление 1): цена кавера Nexus — utilization-формула без
риск-фида; единственный канал репрайсинга под риск — РУЧНОЙ (пул-менеджер поднимает
`targetPrice` или выдёргивает stake). Edge жив, только если менеджеры медленные.

**Измерение №1 (это и есть Gate 1):** как быстро после публичного риск-события
менеджеры реально двигали `targetPrice` затронутых продуктов? Медиана в днях — edge
жив; в часах — мёртв. Плюс глобальная база: как часто targetPrice вообще двигается.

## Источник истины

Событие `ProductUpdated(uint productId, uint8 targetWeight, uint96 targetPrice)` —
эмитится StakingProducts при КАЖДОМ ручном изменении цены/веса продукта менеджером
(`emit` в `setProducts`, StakingProducts.sol:295, master). Оно же ловит и «мягкое
делистинг» (targetWeight → 0). Всё остальное (bumpedPrice) — автоматика спроса,
менеджера не характеризует.

- Топология: один прокси-контракт на все пулы → один адрес для скана.
  ВАЖНО: логи эмитятся StakingProducts (общий), а не пер-пул минипрокси; `poolId`
  в событии НЕ индексирован и вообще не входит в поля — атрибуция к пулу через
  `tx.to`/трейс не нужна для Gate 1 (нам важен факт и время репрайсинга продукта,
  не автор). Если пригодится пул — восстановим по `txHash` → `eth_getTransactionByHash`
  (метод `setProducts(uint poolId,...)` первым словом calldata).
- Прокси апгрейдился → исторические имплементации могли эмитить ДРУГИЕ события.
  Скан по адресу БЕЗ topic-фильтра, группировка по topic0, распознаём известные,
  неизвестные — в отчёт (не терять молча).

## RPC-план (замерено 2026-07-05)

- Primary: `https://gateway.tenderly.co/public/mainnet` — getLogs 100k-окно, ~0.2 с.
- Fallback: `https://rpc.mevblocker.io` — getLogs 100k-окно ОК.
- drpc — только ≤10k-окна; publicnode — 403 на getLogs (как на Base). llama/blockpi лежат.
- Объём: V2 живёт с ~марта 2023 (блок ~16.7M), голова ~25.47M → ~90 запросов
  по 100k. Таймстемпы — `eth_getBlockByNumber` только для блоков с матчами.

## Модули (переиспользуем подход monad-liquidator, код самодостаточный)

- `analysis/keccak.py` — копия pure-stdlib Keccak-256 (с тестами-векторами).
- `analysis/rpc.py` — копия read-only JSON-RPC клиента, DEFAULT_RPCS → Ethereum
  (tenderly, mevblocker), whitelist-методы без изменений.
- `analysis/nexus_repricing.py` — сканер + отчёт:
  1. `scan` — все логи StakingProducts с 16_700_000 до головы, кэш в
     `data/staking_products_logs.json` (сырьё: block, ts, topic0, topics, data, tx).
     Инкрементальный дозапуск от последнего кэшированного блока.
  2. `decode` — известные topic0 → записи `{ts, productId, targetWeight, targetPrice}`.
  3. `report` — (а) глобально: продуктов с ≥1 апдейтом, апдейтов/продукт/год,
     гистограмма межапдейтных интервалов; (б) кейсы: для каждого инцидента из
     `INCIDENTS` (имя-матч продуктов по `data/nexus_products.json` из
     api.nexusmutual.io/v2/products) — время первого ProductUpdated ПОСЛЕ T0
     инцидента, латентность в часах/днях.
- Чистые функции (decode, латентность, гистограммы) — офлайн-юнит-тесты, stdlib-only.

## Инциденты для кейсов (T0 = публичное время события, UTC; уточняются при анализе)

V2-эра only (FTX/UST были при V1 — другой прайсинг, в замер не входят):
Euler 2023-03-13, Curve 2023-07-30, KyberSwap 2023-11-22, Bybit 2025-02-21,
Arcadia 2025-07-15, Balancer 2025-11-03, Stream Finance 2025-11-04.
Для каждого кейса фиксируем: какие листинги считаем «затронутыми» (прямой листинг
протокола + смежные, напр. yield-токены на нём) и откуда взят T0.

## Критерий результата (заранее, чтобы не подгонять)

- **Edge жив**: медианная латентность первого репрайсинга затронутого листинга > 48 ч,
  И глобальная частота апдейтов низкая (медиана продукта — реже раза в квартал).
- **Edge мёртв**: медиана < 12 ч или систематические превентивные репрайсинги.
- Между — серая зона: решает форвард-мониторинг (снапшоты /v2/capacity уже начали
  2026-07-05).

## Не-цели этой фазы

- CoverBought/спрос (Cover-контракт) — фаза 2, если латентность не решит вопрос.
- Выдёргивание stake менеджерами (события на минипулах) — фаза 2.
- Никаких транзакций, никаких ключей; только публичные RPC, read-only.
