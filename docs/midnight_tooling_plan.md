## 1. Почему это целевой кандидат под {латентность, капитал десятки $k}

Сводка (всё [P: код], детали в mechanics):
- **Не аукцион** (режим 1 решётки не срабатывает): normal = Blue-подобный fixed-spread FCFS;
  post-maturity = линейный рамп `lif(t) = min(maxLif, 1 + (maxLif−1)·(t−maturity)/3600s)`
  (TIME_TO_MAX_LIF = 60 мин), обычный FCFS-вызов.
- **Расписанное событие**: maturity — поле структуры в `MarketCreated`
  (topic0 `0xdbf3e95a…c9472`) — календарь окон известен ЗАРАНЕЕ. Это тайминг-игра
  (выбор break-even момента входа в рамп), а не реакция на внезапность.
- **Post-maturity RCF отключён** — сейзабельна ВСЯ позиция; `liquidatorGate == 0` ⇒ полностью
  permissionless, включая post-maturity (проверено по строкам 636–637 Midnight.sol).
- **Поле тулинга пусто** (снимок 2026-07-05): официальный бот Blue-only, публичных
  Midnight-ликвидаторов нет, `midnight-sdk` — только offer-флоу.
- Профиль совпадает: подготовленность (календарь+модель) — первична; латентность — тай-брейк
  FCFS-коллизии на границе break-even; капитал десятков $k достаточен для атомарного пути
  (callback отдаёт залог ДО repay — см. §6).

## 3. Тулинг M-T1..M-T5 (спека, бюджеты)

**M-T1. Maturity-календарь** (~0.5–1 д). Инкрементальный сканер `MarketCreated` → реестр рынков
(все параметры из §2.2) + tick-лист maturity-окон, отсортированный по времени. stdlib-only, кэш
`data/midnight_markets.json`, тесты (offline-keccak topic0 уже сверен двумя путями). Выход —
календарь + отчёт по гейтам/оракулам (питает M-T4 и §5).

**M-T2. Break-even-модель рампа** (~1–2 д, ядро). См. §4. Реюз exit-роутера из
`analysis/svr_gate1_replay.py` (Uni V3 перебор fee-комбо + Curve + стейбл-интермедиари; квоты
`eth_call` на текущем блоке) на Base-пулах. Выход: `t*(позиция)` и paper-EV окна; юнит-тесты
на формулу рампа.

**M-T3. Fork-harness** (~1–2 д). Anvil-форк Base против РЕАЛЬНОГО деплоя: warp за maturity,
исполнение `liquidate` с `ILiquidateCallback` (свап залога до repay), проверка liveness-граблей
(реверт оракула любого активированного коллатерала; oracle=0 repaid-путь; «liquidation locked» в
callback `take`). Де-рискает исполнение ДО любого live; на живой сети НИЧЕГО не шлём.

**M-T4. Вотчеры** (~0.5 д). (i) доля gate≠0 (мера огораживания поля); (ii) оракул-фиды рынков
(SVR/Atom-маркер); (iii) Adapter WIP→готов (бутстрап lender-потока = предвестник borrow);
(iv) diff `addresses.ts` (re-deploy-детектор — актуально при DRAFT-аудитах);
(v) audits/ не-DRAFT. Каденс: еженедельно (ускорено с 2 нед — деплой случился).

**M-T5. Shadow-режим** (постоянный, ~0 инженерии поверх M-T1/2). На каждое прошедшее
maturity-окно с ненулевым долгом фиксировать: `t_первого_филла − maturity`, реализованный бонус,
адреса ликвидаторов, наш paper-`t*` и «выиграли бы?». Метрика конкуренции = уникальные
ликвидаторы/окно. Это прямой аналог микротеста контестируемости SVR Gate 1 — только форвардный.

## 4. Break-even-модель рампа (ядро M-T2)

Обозначения: D — погашаемый долг (post-maturity RCF off ⇒ вся позиция), lif(t) как в §1,
Δt = t − maturity.

```
бонус(t)   = lif(t) − 1  =  min(maxLif−1, (maxLif−1)·Δt/3600)
π(t)       = D·(lif(t)−1)                # оракульный gross
           − impact(D·lif(t))            # реальный exit-квотой (роутер), НЕ оракулом
           − gas − flashCost             # затраты пути
t*         = min { t : π(t) > 0 }        # наш break-even
```

Игра FCFS: каждый бот стреляет в свой t*; выигрывает НИЗШИЙ жизнеспособный t*. Наши рычаги
снижения t*: (а) лучший exit-роутер (меньше impact — главный член на клипах), (б) газ-эффективный
путь, (в) точный расчёт maxLif пер-коллатерально. Латентность — тай-брейк при коллизии на границе
t* (Base sequencer = FCFS-по-прибытию). Уроки, встроенные в модель:
- **Exit решает** (SVR Gate 1: оракульная маржа 50% → realized 9% на полном клипе): impact
  считаем квотой реального роутера на размер ВСЕЙ позиции, никогда оракулом.
- Мульти-коллатерал: сейз по нескольким токенам ⇒ impact суммируется по ногам выхода; oracle
  liveness ЛЮБОГО активированного коллатерала обязателен (иначе liquidate ревертит) — модель
  должна проверять live-оракулы перед t*.
- Normal-mode (health-based) сегмент наследует Blue-гонку и OEV-угрозу — вне фокуса этого плана
  (это территория живого Base-трека monad-liquidator).

## 5. Pre-registered операционные пороги (фиксация ЭТИМ коммитом, ДО замеров)

Согласованы с pre-reg B @ `673f0d8` (≤3 ликвидатора; ≥$5k/90д; порог живости $10M) — не
ослабляют их, а операционализируют. Разрешённые исходы: monitoring / shadow / handover;
**«pilot с капиталом» из этого плана НЕ назначается** — live-шаг только отдельным решением
пользователя после Gate 1 в треке monad-liquidator (fork-replay), как записано в pre-reg.

- **Активация shadow (M-T5)**: ≥1 рынок с `liquidatorGate == 0` и borrow > 0.
- **Handover в monad-liquidator (на его pilot-гейт)** — ВСЕ одновременно:
  1. ≥3 прошедших maturity-окон с сейзабельным долгом ≥ $5k каждое;
  2. медиана уникальных ликвидаторов на окно ≤ 3;
  3. paper-модель M-T2 при консервативной задержке (наш t* + 2 блока) выигрывает ≥ 1/3 окон;
  4. медианный paper-net окна ≥ 3× газ-затрат; p25 paper-net > 0;
  5. exit: quoted impact p75-клипа ≤ 2% нашим роутером (анти-Monad / анти-SVR-Gate-1 условие).
- **Возврат в monitoring**: 8 недель после первого gate=0 рынка НЕТ ни одного окна с сейзабельным
  долгом ≥$5k; ИЛИ gate≠0 на >80% рынков (по borrow-весу при потоке).
- **Kill (немедленно, любой)**: SVR/Atom-фид на наблюдаемых рынках (OEV-изъятие, режим 1);
  ≥10 уникальных ликвидаторов в окнах первых 2 недель потока (рой пришёл раньше);
  re-deploy с изменённой механикой liquidate (через вотчер M-T4.iv — тогда план на пересборку).

## 6. Роль латентности и капитала (честно)

- **Латентность** — разрешена пользователем; на Base это качество подачи к секвенсору
  (FCFS-по-прибытию). Она НЕ выбирает t* (это модель), но решает коллизию в t*. Согласуется с
  mechanics §5 («латентность НЕ решает» — про выбор момента) и с режимом провала #9 («латентность —
  билет, не доля»): здесь билет дешёвый, а доля определяется моделью+exit.
- **Капитал десятки $k** — достаточен: `ILiquidateCallback` отдаёт залог ДО repay ⇒ атомарный
  путь «свап залога → repay» без предзанятого капитала; flash-loan — запасной; собственный
  капитал = газ + буфер + мелкий инвентарь для не-атомарных ног. Капитал НЕ переворачивает
  exit-неликвидность крупного клипа (урок SVR Gate 1) — поэтому порог §5.5.

## 7. Риски / не верифицировано / оговорки честности

- **Байткод НЕ верифицирован он-чейн** (egress текущей сессии блокирует все Base-RPC и
  эксплореры; свидетельство деплоя = официальный SDK-реестр + бот-коммит). Возможен вариант
  «staging-деплой команды, не прод». Шаг 0.1 обязателен до любых выводов.
- **Аудиты DRAFT** ⇒ ненулевая вероятность re-deploy/патчей; ответ — вотчер M-T4.iv, тулинг
  строить модульно (адрес — конфиг).
- **`enterGate`** может ограничить заведение займов узким кругом, а бутстрап lender-потока
  (Adapter) — WIP: поток может не прийти месяцами. Ответ — порог «возврат в monitoring», не
  ожидания.
- **Рынки могут деплоиться с `liquidatorGate ≠ 0`** (KYC) — эмпирика деплоя, датчик в M-T4.i.
- **Размер приза неизвестен до потока** — не обещаем (антипаттерн «размер приза», Injective).
  Порог §5.4 отсеивает пыль автоматически.
- **Это транзиторное окно, не структурный вечный edge** (исключение (i) закона проекта, как
  Base-тейл): эрозия неизбежна (официальный бот получит Midnight-поддержку; OEV-слой доползёт).
  Цена входа мала (~неделя инженерии), ставка ограничена подготовкой — это осознанная покупка
  временнОго окна, что честно соответствует единственному исторически прошедшему архетип-кейсу.
